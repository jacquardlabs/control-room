# Design: T1 — Read-only control-room

**Date:** 2026-07-11 (revised same day — see revision note)
**Status:** Design, pre-implementation. Recorded now per CLAUDE.md's scope ("backlog
grooming, spec work, and design-doc drafting are in scope now — code is not"). T1's
entry gate is the board-event schema being hardened on a real epic (PRODUCT.md,
founding-note.md addendum 3) — satisfied as of studious's `worker-evidence-and-board`
epic, whose real `/gate-audit`/`/gate-acceptance` cycles caught and fixed real
defects; the gate no longer waits on studious's own board-ui, which continues to
evolve independently (see below).

**Revision note:** this doc originally described adopting studious's Flight Deck
renderer directly. Studious's board-ui redesigned within days of shipping (Flight Deck
→ "Operator Graphic") and renamed its own chrome to "the control-room idiom" —
colliding with this product's name. Corrected here and in PRODUCT.md/DESIGN.md/
founding-note.md (addendum 3, 2026-07-11): control-room adopts the schema only and
builds its own renderer.

**Milestone:** T1 — Read-only control-room (GitHub issues #1–#9)
**Source:** studious #98's design-doc contract, invoked by issue #9 ("Design doc for
T1 (gate-design-review before build)")

## Problem & persona

Primary persona (PRODUCT.md, verbatim): "The owner-operator running multiple Claude
Code work streams daily — epics via `/work-through`, winnow-style product builds in
parallel worktrees, background research fleets, interactive sessions — who currently
discovers 'it needed me an hour ago' by going tab to tab."

The problem: on an ordinary day the owner runs 5+ concurrent streams, observable only
as separate terminal tabs and task counts. The two questions that matter — *is each
stream progressing, and what needs me right now* — have no single answer today;
answering them costs a manual sweep across every tab. The existing PTY-multiplexer
category (herdr/Orca/Conductor) doesn't solve this: it shows *what's typing*, not
*what's progressing* or *what needs a human*.

## Proposed design

T1 delivers control-room as a read-only local web app the owner keeps open on a
second monitor, built from three always-or-often-visible surfaces:

- **The wall (fleet strip)** — always visible: N grinding · R review-ready · M need
  you · MASTER CAUTION with unacknowledged count · aggregate burn. This is the answer
  to "is anything progressing / does anything need me," collapsed to a glance. The
  seven attention states resolve into these counts explicitly, so none is silently
  lost inside another: **N (grinding)** is `grinding` only. **R (review-ready)** is
  `review-ready` only, kept as its own count rather than folded into grinding — a
  finished-and-waiting stream must never look like a still-working one. **M (need
  you)** is `input-blocked` + `question-pending` + `parked` + `died` — `died` counts
  here (a dead stream demands attention as much as a blocked one does) but keeps its
  own red glyph per-stream rather than rendering amber, since DESIGN.md reserves red
  for `died` alone. **MASTER CAUTION blinks only for M**, never for R — review-ready
  is advisory, not urgent, so it's visible but doesn't claim the one motion channel
  reserved for "a human is needed." `done` streams inflate no count; they age out per
  "instruments never move," same as a dead stream, just without the red glyph.
- **Stream tabs** — one per live stream (interactive session, Workflow run, or
  background task), discovered automatically from disk (transcripts, journals, task
  state) with zero instrumentation required of the streams themselves. A tab's badge
  always carries the same attention glyph as its strip entry — strip and tab never
  disagree. Streams that emit the board-protocol (studious epics first) render full
  semantic detail (parks, verdicts, fix-budget pips) through control-room's own
  schema-driven renderer, consuming studious's board schema
  (`board-schema.md`/`events-format.md`) as data — not studious's rendering code;
  streams with no protocol render generic vitals (progress, cost, attention state)
  through that same renderer — one renderer control-room owns, two data richness
  levels, never a fork in the rendering code and never a second visual idiom inside
  control-room's own surface.
- **Notifications + acknowledge** — an OS notification fires only when a stream's
  attention state *changes* (not on every tick); the owner acknowledges from the wall
  or a tab, which stops the blink but leaves the record (CAS/drawer) intact.
  Unacknowledged needs-you is the only thing that moves repeatedly on any surface.
  **Detection freshness is the mechanism that makes the exit gate achievable, not an
  afterthought:** the primary detection path for a state change is Claude Code's own
  hook mechanism, firing synchronously at the same moment the harness's own
  notification would — not disk-tail/journal polling alone, which is bounded by a
  multi-second poll interval and could otherwise land behind the very harness
  notification T1 exists to beat (PRODUCT.md: "it earns that spot only if it surfaces
  needs-you moments faster than the harness's own notifications"). Disk-tail polling
  remains the detection path for streams that don't or can't fire hooks (background
  tasks, and reconciliation on restart) — but for a live interactive session or
  Workflow run, the hook path is what makes "faster than the harness" a design
  commitment rather than a hope.

Leans on PRODUCT.md's principles 1 (progress-and-attention, not logs — logs stay one
click away in the per-stream drawer, never ambient), 2 (blink reserved for
needs-a-human), 3 (instruments never move — new streams append, dead ones gray then
age out), 4 (read-only through T1 — the tool renders disk truth and writes nothing but
its own local ack/prefs), 5 (the operator acts, the tool never does — T1's read-only
posture and its "no auto-anything" scope both stem from this), 6 (local and keyless —
no hosted service, transcripts never leave the machine), and 7 (wraps whatever
workflow — attention semantics arrive by protocol, the generic detector covers
everything else).

**The anti-false-amber invariant is load-bearing, not a tuning goal.** PRODUCT.md
names false needs-you as the product-killing failure mode (known problem #2): every
detector's uncertainty degrades to `grinding`, never to a false amber. This design
treats that as a hard invariant the build must satisfy, not an aspiration to tune
toward later — every detector ships with golden-transcript fixtures per attention
state before T1 exit (CLAUDE.md), including adversarial fixtures built from
genuinely ambiguous transcript tails, whose required output is `grinding`. A detector
that cannot be shown to degrade correctly on an adversarial fixture is not done.

Four decisions this design commits to: board-protocol streams are rendered by
**control-room's own renderer, against studious's adopted schema** — not a vendored
copy of studious's board-ui (settled 2026-07-11 — see revision note above and
founding-note.md's addendum 3); **attention-state detection is hook-first, poll-
fallback** (above) — the design commits to this now because detection architecture is
expensive to retrofit and the exit gate is a timing bar, not a features bar; the
transcript-parsing code shared with cctx is **vendored into control-room as a tracked
copy** (upstream fixes flow in by update, not by fork — this vendoring decision is
unaffected by the renderer correction above, since parsing code doesn't carry a
competing product's visual identity the way a renderer does); and the wall carries an
**explicit liveness indicator** (last-updated/connection state) so a silently stalled
server or dropped SSE connection is visibly degraded chrome rather than a frozen
screen that reads as false-calm "grinding."

## User journey

Touches PRODUCT.md's critical journey #1, "The glance," almost verbatim, and journey
#2, "The interrupt done right." Nothing about the owner's *existing* workflow
changes — control-room T1 is a new vantage point, purely additive.

1. Mid-workday, owner glances at the second monitor. The wall reads 6 grinding, 2
   need-you, MASTER CAUTION blinking with a count of 2. The 6 grinding tabs sit
   motionless — only the caution lamp pulses, per "instruments never move" and
   "blink is reserved."
2. Owner clicks the caution lamp or one of the two amber tabs. If that stream is a
   studious epic, control-room's own board renderer (fed by studious's schema) shows
   exactly which story parked, its verdict token, and the fix-budget pips leading up
   to it. If it's a plain session or task with no protocol, generic vitals show
   `input-blocked` or `question-pending` with its one-clause reason — same glyph
   vocabulary either way, same renderer either way.
3. The drawer (opened on demand, never ambient) shows the verdict trail or transcript
   excerpt and, where one exists, a copy-pasteable resolution command (e.g. a
   `gate-ledger` invocation for a parked studious story).
4. Owner acts in the actual terminal/session — T1 is observe-only, the tool itself
   never acts.
5. Back in control-room, the owner acknowledges: the blink stops, the CAS/drawer
   record stays (ack is local render state, never a ledger write). If a *new* event
   later fires on that same stream, it re-notifies; until then, silence.

Separately, journey #2: owner is deep in unrelated work, gets exactly one OS
notification naming the stream, the state, and the one-clause reason ("auth-refresh
parked — NEEDS DISCUSSION"). They decide in ten seconds whether it can wait — usually
it can — which is the whole point: a decision instead of a tab-sweep-driven anxiety.

## Out of scope

Cross-checked against PRODUCT.md's "What we're NOT building" and the T2/T3/T4
milestone boundaries:

- **Any operator action from the app** (T2) — resolving a park, answering a question,
  permission decisions. T1 stays strictly observe-only; the owner still acts in the
  terminal.
- **Launching anything from the app** (T3) — epics, workflows, sessions, presets,
  worktree lifecycle.
- **The desktop appliance and Claude credentialing** (T4) — no packaged app, no Agent
  SDK auth; T1 makes no model calls of its own.
- **The per-epic board's standalone form** (`studious board <slug>`) — ships in
  studious (#98); control-room adopts its schema, not its renderer or code, and
  renders board-protocol streams with its own renderer (see revision note above).
- **Session analytics/diagnosis** — cctx's lane. T1's cost vitals are current-state
  numbers only (per issue #7: "no thresholds, no warnings").
- **Ambient log panes or streaming text anywhere outside the drawer** — the founding
  anti-pattern, enforced structurally.
- **Auto-anything** — no auto-answer, auto-approve, auto-retry, policy engine, even in
  service of "smarter" rendering.
- **Multi-user, hosted, or telemetered anything** — local-first and keyless
  throughout.
- **Non-macOS notification delivery** — T1 targets `osascript` first; other platforms
  aren't committed to this milestone (see Open questions).

## Alternatives considered

1. **PTY/log-pane multiplexing** (herdr/Orca/Conductor's approach — panes of
   scrolling agent terminals). Rejected: this is the founding anti-pattern itself. It
   answers "what is it typing," not "is it progressing / does it need me." The timed
   comprehension test (design-history.md) is the empirical receipt — four boards
   tested across randomized failure scenarios, and the denser, more log-like
   alternatives (Flow Board, Transit Map, a plain Control Room console) lost to Flight
   Deck on time-to-last-correct-answer and wrong-click count.
2. **Terminal-only renderer (TUI, not a web app)** — considered explicitly during the
   same design exploration and rejected: "kickback arcs, swimlane density, and the
   fan-in drawer don't fit a TTY; revisit only as debug tooling." A TUI can't render
   the fix-budget wedges or drawer-level copy-paste resolution commands at the
   fidelity the comprehension test validated.
3. **Fork studious's board schema instead of adopting it** (define control-room's own
   independent attention/board-event taxonomy) — rejected per CLAUDE.md's "Adopt,
   don't fork." Two independently-defined schemas for the same concept is exactly the
   "one-emitter spec-graveyard risk" the founding note names as the reason to adopt
   instead — schema drift here would be silent and cumulative in a way this design
   doc couldn't catch at review time.
4. **Vendor studious's board-ui renderer wholesale** (the original plan this doc
   committed to, before this revision) — rejected on evidence, not just principle:
   studious's board-ui redesigned entirely within days of shipping (Flight Deck →
   Operator Graphic, studious PR #127) and renamed its own chrome to "the control-room
   idiom," colliding with this product's name. A vendored renderer from a separately-
   evolving product churns on that product's timeline, for that product's reasons —
   the opposite of the stability "adopt, don't fork" is meant to buy. It would also
   mean two visual idioms inside control-room's own surface (studious's for epic tabs,
   control-room's own for every other tab), which cuts against "one vocabulary across
   the fleet" more than it serves it.
5. **Embed/deep-link studious's own running `board-server` UI per epic tab** (e.g. an
   iframe) instead of rendering the schema in control-room's own chrome — rejected:
   this has the same visual-inconsistency problem as vendoring, worse, and adds a
   process-lifecycle dependency (studious's server must be running per epic, it's an
   opt-in `--open` tool, not always-on) control-room otherwise doesn't need.
6. **Extract a real shared library for the cctx-overlap parsing code**, rather than
   vendoring a tracked copy — simpler in principle (one source of truth, no manual
   sync) but rejected for T1: it adds a cross-repo release/versioning step and likely
   a dependency, in tension with the stdlib-only, no-build-step convention both
   viva-pattern tools already follow. The tradeoff accepted instead (manual sync,
   drift risk) is named explicitly below.

## Operational readiness

Mostly N/A for a local, single-user, keyless tool — stated explicitly rather than
omitted:

- **Migration & rollback**: N/A — no operational surface. control-room holds no
  persistent state beyond a local ack/preferences file, itself disposable: killing
  and restarting the server reconstructs the whole screen from disk truth (issue #4's
  acceptance criteria). Nothing to migrate; "rollback" is stopping the process.
- **Rollout**: N/A — no deployment, no user population beyond the owner running it
  locally.
- **How you know it's working vs. failing**: the wall's liveness indicator (decided
  above) turns a silent stall into visibly degraded chrome instead of a
  frozen-but-plausible screen. Beyond that, issue #8's dogfood week *is* the
  production-readiness instrument here — a real multi-stream week ending in a written
  keep/kill verdict against the exit gate, which is the only "monitoring" a tool with
  one user needs.

## Open questions

- **cctx vendoring sync cadence** — direction is set (vendor a tracked copy),
  mechanism isn't: what triggers a re-sync, who notices drift between the two copies.
- **Multi-instance behavior** — undefined: two machines watching the same worktrees,
  or two browser tabs against one server. Not addressed anywhere yet; probably fine to
  assume single-operator/single-machine, but that assumption isn't written down
  elsewhere.
- **Liveness indicator's exact visual treatment** — that it exists is decided here;
  how it renders belongs to control-room's own renderer and DESIGN.md's conventions —
  deferred to plan phase.
- **Notification platform scope beyond macOS** — issue #6 says "osascript first,"
  implying room to grow, but this doc doesn't commit T1 to a second platform. Worth an
  explicit call before it's assumed either way.
- **Renderer parity effort for board-protocol streams** — control-room now owns
  reproducing the semantic fidelity (parks, verdicts, fix-budget pips) that studious's
  board-ui already renders, from the schema alone. Studious's `assets/board-ui/`
  source remains a useful *reference* to read for how it maps schema fields to
  meaning — not to vendor — while control-room builds its own; scoping that effort is
  a plan-phase question, not decided here.
- **Notification debounce for rapid state oscillation** — a detector flickering near
  a heuristic threshold (e.g. grinding → amber → grinding → amber within a short
  window) could fire a burst of near-duplicate notifications for what is really one
  continuous state, eroding the "ten-second decision, not anxiety" trust journey #2
  promises. Some debounce/hysteresis window is needed; its exact threshold is a
  plan-phase tuning question, not decided here — flagged now because it's the same
  trust surface the anti-false-amber invariant protects.
- **control-room's own wall/fleet-strip visual language is also Flight-Deck-descended**
  from the same comprehension test studious's board originally shared, before
  studious's later, differently-tested "Operator Graphic" direction. Whether
  control-room's own look deserves a fresh look given that newer evidence is a
  separate design question, deliberately not resolved by this doc (see founding-note.md
  addendum 3).
